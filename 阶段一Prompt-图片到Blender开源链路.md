# 阶段一执行 Prompt：图片到 Blender 的全开源视觉资产链路

你是一名资深 Python、三维资产管线和 Blender 自动化工程师。请在当前代码仓库中完成阶段一任务，并对所有实际结果负责。不要只给方案；应在不破坏现有用户修改的前提下完成实现、运行验证、记录失败，并交付可重复执行的命令和产物。

## 一、阶段目标

使用开源或许可证明确、允许当前项目使用的组件，打通以下最小垂直链路：

```text
单物体参考图片
→ 图片预处理
→ TripoSR 生成原始三维网格
→ Blender 官方版本通过 Python API 后台导入
→ 统一坐标、尺度、对象命名和基础材质
→ 导出标准 GLB
→ 重新导入并验证
→ 生成可视化预览和资产报告
```

本阶段的核心目标是证明：给定一张合格的单物体图片，可以通过一条可重复执行的开源工具链，得到 Blender 可读取、项目可管理、后续可以继续物理资产化的标准视觉资产。

## 二、强制技术边界

1. 图生三维首选 TripoSR。代码、模型权重和依赖的许可证必须分别核实并记录，不能只看到 GitHub 仓库公开就认定全部可商用。
2. 场景处理必须使用 Blender 官方开源版本及其内置 Python API。
3. 不使用 DesignToolbox，不调用闭源图生三维 API，不依赖 Blender MCP。
4. Blender 自动化必须支持 `--background --python`，不能要求用户逐个手工点击才能完成主流程。
5. 内部处理可以使用许可证明确的开源 Python 库，例如 Pillow、NumPy、trimesh 和 Open3D；引入前记录版本和许可证。
6. 不允许把大模型生成的几何参数当作测量结果。本阶段的大模型只负责辅助开发，不负责虚构模型尺寸、质量或物理属性。
7. 不得提交 API key、登录信息、模型缓存、大型权重或不明确授权的素材。
8. 不得静默切换到商业服务。若本机硬件不能运行 TripoSR，应输出证据和回退建议，等待用户授权后再使用远程 GPU。

## 三、本阶段非目标

本阶段不实现以下内容：

- 房间多视角自动重建；
- 一张图片中多个物体的自动拆分；
- 碰撞体生成；
- 质量、质心、惯量、摩擦等物理参数；
- 柜门、抽屉等关节；
- 抓取点和机器人任务；
- 任意模型的自动拓扑修复；
- 对视觉还原精度作没有真实基准的承诺。

这些内容只能记录为后续阶段输入，不能为了看起来“完整”而提前伪造。

## 四、输入要求

输入图片必须满足：

- 主要内容只有一个完整物体；
- 背景尽量纯净，物体边界清楚；
- 无明显遮挡、截断、水印和强烈运动模糊；
- 建议分辨率为 512×512 或 1024×1024；
- 首个测试对象优先使用纸盒、无把手杯子等简单刚性物体；
- 若图片包含多个物体，应停止并报告，或先生成明确的单物体裁剪图；
- 若有真实尺寸，应记录为米，例如宽 0.30 m、高 0.12 m、深 0.20 m；不得从图片中假装精确测量。

开始执行前，检查用户提供的图片。仓库中现有的多几何体组合图片不能直接作为首个单物体测试输入，除非先明确裁剪出唯一目标。

## 五、环境检查

先进行只读检查并记录：

- 操作系统、CPU 架构、内存；
- 是否为 Apple Silicon；
- Python 版本和当前虚拟环境；
- PyTorch 是否可用，以及 CUDA、MPS 或 CPU 后端情况；
- Blender 是否安装、准确版本、可执行文件路径；
- 可用磁盘空间；
- 仓库当前 Git 状态，避免覆盖用户已有修改；
- 项目现有依赖管理方式、测试方式和目录约定。

如果 Blender 未安装，给出官方开源版本的安装方案。安装或下载大型模型权重前，先说明预计下载量、磁盘需求和硬件风险。不要声称 Apple MPS 一定兼容；必须以实际运行结果为准。

## 六、目录与产物规范

遵循仓库现有约定；若没有对应目录，建议采用：

```text
assets/<asset_id>/
├── source/
│   └── reference.png
├── preprocessed/
│   ├── foreground.png
│   └── mask.png
├── generated/
│   ├── raw.obj 或 raw.glb
│   ├── raw.mtl
│   └── textures/
├── blender/
│   └── normalized.blend
├── export/
│   └── <asset_id>.glb
├── preview/
│   └── turntable.png 或 preview.png
├── logs/
│   ├── generation.log
│   └── blender.log
└── asset_manifest.json
```

原始输入和 TripoSR 原始输出必须保持不可变。清理、缩放和导出的文件放入独立目录，不能覆盖原文件。

## 七、详细执行步骤

### 步骤 1：建立资产清单

为测试对象确定稳定的 `asset_id`，保存原图副本并计算文件哈希。创建 `asset_manifest.json`，至少记录：

- `asset_id`；
- 原始图片路径与哈希；
- 图片尺寸；
- 目标物体名称；
- 用户提供的真实尺寸及来源；
- TripoSR 代码版本、权重版本和许可证；
- Blender 版本；
- 生成参数和随机种子；
- 生成、标准化和验证状态；
- 所有警告、假设和失败原因。

### 步骤 2：图片预处理

检查物体是否完整、是否为单一主体。按 TripoSR 的实际输入要求完成缩放、色彩模式转换和必要的去背景。保留透明前景图与掩码，不覆盖原图。

不要为了改善结果而对图片做不可追踪的手工修改。每项处理都必须能通过命令或脚本重复执行，并记录参数。

### 步骤 3：安装并验证 TripoSR

使用独立、可复现的 Python 环境。锁定代码提交或发布版本及依赖版本。先运行官方最小示例，再运行项目输入。

按以下优先级选择计算后端：

```text
官方明确支持的本机 GPU
→ 经实测可工作的 MPS
→ CPU 小规模验证
→ 报告本机阻塞并申请使用远程开源 GPU 环境
```

不得通过修改代码跳过错误后宣称成功。发生不支持的算子、内存不足或输出空网格时，保存完整错误和环境信息。

### 步骤 4：生成原始三维资产

使用固定随机种子和显式参数运行生成。保存标准输出、错误日志、耗时、峰值内存和所有原始文件。不得只保存截图。

生成后立即进行机器检查：

- 文件存在且非空；
- 至少包含一个 Mesh；
- 顶点和三角面数量大于零；
- 顶点坐标均为有限数，不含 NaN 或 Infinity；
- 包围盒三个方向均为正值；
- 记录对象数、Mesh 数、顶点数、三角面数、材质数和纹理文件数；
- 检查纹理路径是否缺失；
- 记录网格是否封闭、自交和法线异常，但本阶段不强制修复所有拓扑问题。

### 步骤 5：编写 Blender 后台标准化脚本

实现可通过命令行调用的 Blender Python 脚本，至少支持：

- 输入 OBJ 或 GLB；
- 清空默认场景；
- 导入所有相关 Mesh 和材质；
- 将对象组织到稳定命名的 Collection；
- 统一对象和 Mesh 命名；
- 保持右手坐标系，场景使用米，竖直方向为 `+Z`；
- 把物体底部移动到 `z=0`；
- 把水平中心移动到世界原点附近；
- 记录导入前后的变换矩阵和包围盒；
- 若用户提供真实尺寸，默认只按可信的一个基准尺寸做等比缩放，不能为迎合宽高深而进行会扭曲物体的非等比缩放；
- 保存标准化 `.blend`；
- 导出 GLB；
- 输出结构化检查报告和退出码。

不要用“应用所有变换”掩盖坐标问题。每次变换的原因和数值必须可追踪。

### 步骤 6：Blender 导入、标准化和导出

使用类似下面的后台命令执行，具体参数以实际脚本为准：

```bash
blender --background \
  --python scripts/blender/normalize_asset.py \
  -- \
  --input assets/<asset_id>/generated/raw.obj \
  --output assets/<asset_id>/export/<asset_id>.glb \
  --blend-output assets/<asset_id>/blender/normalized.blend
```

命令必须可以从干净终端重复运行。失败时返回非零退出码，不得生成一个看似正常但内容为空的 GLB。

### 步骤 7：往返验证

新建一个空 Blender 场景，重新导入最终 GLB，不依赖原始 `.blend` 中的状态。验证：

- GLB 能无报错导入；
- 至少有一个可见 Mesh；
- 对象、材质和纹理没有全部丢失；
- 包围盒与导出前一致，允许浮点和格式转换造成的小误差；
- 底部位于 `z=0` 附近；
- 若给定真实基准尺寸，基准方向误差不超过 1%；
- 场景没有极端尺度，例如毫米物体被解释为数十米；
- 重新导入后对象不是全黑、全透明、空白或位于相机不可见处。

### 步骤 8：生成视觉预览

使用 Blender 创建中性背景、基础灯光和固定相机，生成至少一张预览图。预览只用于检查，不允许代替网格验证。

检查预览图不是纯白、纯黑或透明空图，物体完整处于画面内，没有明显被地面截断。保存生成命令和 Blender 日志。

### 步骤 9：提供统一入口

提供一个项目级命令或脚本串联预处理、TripoSR 推理、网格检查、Blender 标准化和往返验证。它应支持：

- 指定输入图片和 `asset_id`；
- 可选的真实基准尺寸；
- 失败即停止；
- 可重复运行；
- 已存在原始输出时不静默覆盖；
- 输出最终产物位置和简明结果摘要。

### 步骤 10：测试和文档

至少为不依赖 GPU 的部分添加自动测试，包括参数解析、清单读写、非法空网格拒绝、包围盒计算和 Blender 脚本的关键纯函数。若环境允许，增加一个小型端到端测试；否则明确记录未运行原因。

文档必须包含安装、输入要求、运行命令、产物解释、常见失败、清理缓存方式和许可证说明。

## 八、验收标准

只有同时满足以下条件，阶段一才算完成：

1. 使用的是许可证已记录的 TripoSR 版本和 Blender 官方版本，没有 DesignToolbox 或闭源生成 API。
2. 一条明确命令可以从合格单物体图片生成最终 GLB。
3. 原始输入、原始生成结果、标准化结果和最终导出结果彼此分离。
4. TripoSR 输出包含有效的非空网格，不含非有限顶点坐标。
5. Blender 后台模式可以无人工点击地导入、标准化、保存 `.blend` 并导出 GLB。
6. 最终 GLB 可以在新空场景中重新导入。
7. 坐标和单位约定明确：米、右手坐标系、`+Z` 向上。
8. 物体底部在 `z=0` 附近，尺度不存在明显数量级错误。
9. 材质和纹理状态被实际检查；若模型本身未生成纹理，应明确报告，而不是伪造“纹理正常”。
10. 生成有效的非空预览图。
11. `asset_manifest.json` 包含版本、参数、来源、许可证、指标、假设和警告。
12. 所有实际运行过的测试通过；没有运行的测试必须说明原因和剩余风险。

以下情况不能判定为完成：

- 只有安装说明，没有实际生成资产；
- 只有 TripoSR 输出，没有成功进入 Blender；
- 只能通过 Blender GUI 手工完成；
- 最终 GLB 未做重新导入测试；
- 图片包含多个主体但仍把混乱结果算作成功；
- 因硬件阻塞而改用未授权商业 API；
- 只提供渲染截图，没有可检查的网格文件和日志。

## 九、重点注意事项

- Apple Silicon 和 MPS 兼容性必须实测，不能依据 PyTorch 支持 MPS 就推断 TripoSR 全部算子可用。
- OBJ、glTF 和 Blender 的坐标、单位及材质表达不同，导入器可能自动旋转或缩放；必须比较变换矩阵和包围盒。
- GLB 可携带纹理，但 OBJ 通常依赖 MTL 和外部图片；移动文件时必须保持引用完整。
- 图生三维模型经常存在背面失真、孔洞、自交、薄面和错误法线。本阶段记录问题，不把视觉可见等同于拓扑合法。
- 不得把生成模型的默认尺寸当成真实尺寸。
- 不得把视觉模型直接当作后续碰撞模型；碰撞简化属于下一阶段。
- Blender 是 GPL 开源软件，但模型权重、输入图片、生成资产和第三方插件各自有独立许可证。
- 大文件和模型缓存应放入 Git 忽略范围；源码、配置、清单和小型测试夹具才进入版本控制。
- 保留完整日志和失败样本，因为后续切换到 InstantMesh、TRELLIS 或其他后端时需要做公平对比。

## 十、失败回退策略

按以下顺序处理失败：

1. 输入不合格：要求提供或裁剪单物体图片。
2. 去背景失败：保留原图并允许用户提供人工确认的掩码。
3. TripoSR 本机后端不兼容：尝试官方支持配置，再评估 CPU 最小验证。
4. 内存或速度不可接受：报告资源需求，申请使用 Linux + NVIDIA GPU 运行同一开源代码。
5. 输出网格为空或严重破碎：保存失败样本和参数，不进入 Blender 并返回失败。
6. Blender 导入失败：判断是格式、纹理路径、插件还是版本问题，优先转换为标准 OBJ/GLB，不依赖 GUI 修补。
7. TripoSR 质量不满足后续要求：在保持统一输入输出协议的前提下，再评估 InstantMesh 或 TRELLIS；不得直接重写整个管线。

## 十一、最终汇报格式

完成后按以下顺序汇报：

1. **结论**：阶段一成功、部分成功或失败。
2. **实际链路**：列出真正运行的组件、版本和计算后端。
3. **运行命令**：给出从图片到最终 GLB 的准确命令。
4. **输出产物**：列出原始网格、`.blend`、GLB、预览、清单和日志路径。
5. **量化结果**：耗时、顶点数、三角面数、对象数、材质数、包围盒和最终尺寸。
6. **验证结果**：逐项对应验收标准，不能只写“测试通过”。
7. **已知问题**：纹理、背面、拓扑、尺度和硬件限制。
8. **许可证**：分别说明代码、权重、素材和输出资产的许可证状态。
9. **下一步**：只有阶段一验收通过后，才建议进入碰撞几何和物理资产化阶段。
